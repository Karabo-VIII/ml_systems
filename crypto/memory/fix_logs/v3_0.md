## 2026-04-07: --clean argparse + clean-variant log_vars IndexError
**Severity:** High (blocked f34 training start)
**Files:** `src/wm/v3/v3_training/train_world_model.py`
**Commits:** 8371b2f, 5f25f97

### Bug 1: --clean flag rejected by main parser (8371b2f)
**Problem:** Import-time `_pre.parse_known_args()` at train_world_model.py:43-44 consumes `--clean` to select the WaveNetCleanModel class, but `parse_known_args()` does NOT strip consumed args from `sys.argv`. Later, the strict main `parser.parse_args()` at line 776 crashed with `unrecognized arguments: --clean`.
**Fix:** Added no-op `parser.add_argument("--clean", action="store_true")` at line 775 so the main parser accepts the flag harmlessly after the pre-parser has already acted on it.

### Bug 2: log_vars layout mismatch for clean variant (5f25f97)
**Problem:** Epoch-banner logging at line 588-596 hardcoded full-RSSM log_vars layout `[rec, kl, ret_1..ret_4, regime]` (7 elements) and read `_s[2+len(REWARD_HORIZONS)] = _s[6]`. But `WaveNetCleanModel.log_vars` is 5-element `[ret_1, ret_4, ret_16, ret_64, regime]` — no rec/kl. Read raised `IndexError: index 6 is out of bounds for dimension 0 with size 5` at epoch 1.
**Fix:** Branch on `_s.numel() == len(REWARD_HORIZONS) + 1` to detect clean layout and use regime index `len(REWARD_HORIZONS)` instead of `2 + len(REWARD_HORIZONS)`.
**Note:** `get_loss()` in world_model_clean.py was already correct (uses `s[hi]` + `s[-1]` position-relative indexing). Bug was purely in the cosmetic epoch-banner print.

### Bug 3: Runner f25 taxonomy mismatch (8371b2f)
**Problem:** `src/run_all_training.py` listed V3 as supporting features `[13, 25, 34]`, but V3 lives in the 41-feature SOTA taxonomy where f25 is NOT a defined slice (V3 settings.py only defines f13/f18/f30/f34/f37/f41). f25 was a copy-paste artifact from V11-V14's hand-picked 3-way taxonomy.
**Fix:** Runner now declares V3-clean as `[13, 34]`.

### Bug 4: Validation gate false-fail on clean (a85c4f0)
**Problem:** `check_gate()` at train_world_model.py:261-268 applied `rec > GATE_REC_MSE_MAX`, `kl < GATE_KL_MIN`, and `kl > GATE_KL_MAX` unconditionally. WaveNetCleanModel has NO recon decoder and NO RSSM KL — both are 0 by construction. `val_metrics.get("kl", 0) = 0 < 0.01` always fails with "KL=0.0000 too low (collapse)". Observed in user's V3-clean f34 run at epoch 5: gate-fail every epoch while training loss was actually decreasing (21.4 → 15.7) and IC was genuinely non-zero. Fix uses `_pre_args.clean` module-level global to skip architecture-absent gates.

### Bug 6: VIB insufficient — switch to full V3 RSSM model (commit 7f9b35b)
**Severity:** CRITICAL — runner now uses full V3, not clean
**Files:** `run_all_training.py` (MODELS entry), `src/wm/v6/v6_training/world_model.py` (import fix)

**Problem:** VIB (commit 9973ca0) did not fix memorization. Re-run 2026-04-08 epoch 10:
  IC1=0.2780, IC4=0.5197, IC16=0.7668, IC64=0.7198, ShIC=0.0001
  Same signature as the pre-VIB run. VIB_KL_WEIGHT=0.05 contributes ~1.4% of
  total loss at max anneal — too weak to counter 4.7M WaveNet params with no
  hard information rate limit. Also GN:inf on 7/10 epochs (WaveNet gated
  activation instability under mixed precision).

**Fix:** Runner now calls `src/wm/v3/v3_training/train_world_model.py` WITHOUT
`--clean`. This uses the full V3 model that already exists:
  - RSSM categorical latent (24x24 = 576 states, log2 ~ 9.2 bits/timestep HARD cap)
  - Gumbel-Softmax straight-through
  - free_nats KL floor (WM_FREE_NATS=1.0)
  - Reconstruction decoder
  - Full ATME (zeros h_seq AND forces obs-only posterior readout)

dual-import pattern matching V3 full.

**Lesson:** When an experimental "clean" variant removes an information
bottleneck, adding VIB back is not sufficient if the KL weight is calibrated
for V1-style small-scale losses. RSSM categorical with free_nats provides a
HARD information cap (you cannot transmit more than log2(classes) bits per
timestep); VIB Gaussian is SOFT pressure that the model can pay through.

### Bug 5: Catastrophic memorization confirmed + VIB fix attempt (9973ca0)
**Status:** Superseded by Bug 6 (runner now uses full V3, not clean).
VIB code remains in world_model_clean.py as research machinery but is unused.

**Severity:** CRITICAL — clean variant unusable without bottleneck
**Files:** `world_model_clean.py`, `train_world_model.py`, `settings.py`

**Confirmed at epoch 10 (2026-04-08):** IC1=0.2729, IC4=0.5104, IC16=0.7440, IC64=0.7112, ShIC≈0.0002. Ratio 0.0007. 1000x worse than V1 healthy (ShIC=0.030).

**Root cause:** WaveNetCleanModel stripped V1.0's RSSM categorical latent (24×24=576 states with KL-to-prior — the load-bearing anti-memorization mechanism) but kept the "ATME" name. ATME implementation `feat = h_seq * atme_mask` is a no-op channel-zeroing applied AFTER the WaveNet TCN has already encoded temporal patterns. 60% of samples see pristine features. With 4.7M params, no information rate limit, and a 31-bar TCN receptive field, the model trivially memorizes (ret_t = f(obs_{t-K:t})).

**Fix:** Variational Information Bottleneck (VIB) replacing the no-op ATME:
- After WaveNet, project h_seq to (mu, logvar) of z_dim=32 latent (vs d_model=256, 8x compression)
- Sample z~N(mu,σ) at training, z=mu at eval
- KL(q(z|h)||N(0,I)) regularizer with VIB_KL_WEIGHT=0.05, annealed via KL_ANNEAL_EPOCHS=20
- Return heads read z_expand(z), NOT h_seq directly
- Regime head still reads h_seq directly (auxiliary slow signal, doesn't conflict with bottleneck)

**Settings added:** VIB_Z_DIM=32, VIB_KL_WEIGHT=0.05, VIB_LOGVAR_INIT=-4.0 (initial std~0.135), VIB_LOGVAR_MIN=-6, VIB_LOGVAR_MAX=2

**Smoke verified:** 4.766M params, KL=2.42 at init, eval mode produces z=mu deterministically, gradients flow through to_mu/to_logvar/wavenet.

**Open question (to be answered by next training run):** does VIB push ShIC ≥ 0.020 with IC1 in 0.04-0.08 range (V1-comparable health)? If not, escalation options: (1) reduce z_dim to 16, (2) increase VIB_KL_WEIGHT to 0.1, (3) reduce wavenet depth/channels, (4) add input-level temporal block shuffle.

## 2026-03-17: ATME z_post leakage + undertraining + Huber shortcut fix
**Severity:** Critical (multiple compounding issues)
**Files:** 16 files across all V3 variants (settings.py, world_model.py, ncl_model.py, train_world_model.py)

### CRITICAL-1: WM_STEPS_PER_EPOCH undertraining
**Files:** `settings.py` (all V3 variants)
**Problem:** WM_STEPS_PER_EPOCH=300 vs V1.0's 2000. 6.7x less data exposure per epoch.
ShIC checks fire before model learns anything. Same bug found in V6 previously.
**Fix:** 300 -> 2000. Also fixed DIVERSITY_STEPS_PER_EPOCH: 300 -> 2000.

### CRITICAL-2: ATME temporal leakage through posterior
**Files:** `world_model.py`, `ncl_model.py` (all V3 variants)
**Problem:** ATME zeros h_seq in feat_heads, but z_post = f(posterior(h_seq, obs_seq)) already
carries GRU's temporal encoding. Zeroing h_seq after the posterior is cosmetic -- temporal
info has already leaked through z_post. Same bug as V4 (less severe for GRU than Mamba
due to lower-bandwidth recurrent state).
**Fix (two-part):**
1. When ATME fires (40% of steps): compute obs-only posterior (no h_seq) for return/regime heads
2. Normal mode: use h_seq.detach() -- GRU learns from recon/KL only, return heads READ
   temporal features but can't OPTIMIZE GRU for memorization

### HIGH-1: DIRECT_RETURN_WEIGHT too high
**Files:** `settings.py` (all V3 variants)
**Problem:** DIRECT_RETURN_WEIGHT=3.0 creates Huber regression shortcut that bypasses
TwoHot discretization bottleneck. Combined with WaveNet's multi-scale temporal features,
accelerates memorization.
**Fix:** 3.0 -> 1.0

### HIGH-2: ATME strength too weak
**Files:** `settings.py` (all V3 variants)
- `TEMPORAL_CTX_DROP`: 0.15 -> 0.40
- `SEQ_SHUFFLE_PROB`: 0.20 -> 0.30
